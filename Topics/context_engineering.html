Two worlds of AI coding

Greenfield

Fresh codebase, clean slate. AI agents perform well — clear context, minimal legacy complexity.

Brownfield

Legacy codebase. AI struggles without guidance — produces churn, "slop", and costly rework without proper context engineering.

The core insight

LLMs are stateless and nondeterministic. Their output quality is entirely determined by what's in the context window. Too small = missing info. Too large = the "dumb zone". The craft is in engineering the middle ground.

Both talks converge on the same principle: humans remain essential. AI amplifies thinking — it doesn't replace it. Bad research or plans cascade into broken code and broken agents.

The three zones

✓

Smart zone (~40% or less)

Focused, relevant context. Model output is precise and reliable. This is the target.

!

Attention fade (40–70%)

Model starts losing track of early context. Output quality becomes inconsistent.

✗

Dumb zone (70%+)

Context overload. The model ignores or contradicts key instructions. High rework cost.

Sub-agents for context isolation

For large codebases, use sub-agents to operate on focused subsystems (frontend, backend, QA). Each sub-agent runs in its own context window and returns a compressed summary to the parent — keeping the parent lean.

Don't anthropomorphize sub-agent "roles." Think of them as isolated context scopes that gather and compress information, not as teammates with personalities.

Why three phases?

Doing research, planning, and implementation in one pass bloats the context window and mixes concerns. Separating phases keeps each step focused and produces a reviewable artifact at each stage.

Phase 1New context window

Research

Investigate and understand the relevant parts of the system. The goal is a compressed research artifact — not a full codebase dump.

1

Scope the task

Identify which files, modules, and services are actually involved.

2

Trace dependencies

Map the call flow, data flow, and any shared state. Use sub-agents for isolated subsystems.

3

Compress findings

Write a concise research file: relevant file paths, key line ranges, dependency summary. Keep it under ~15k tokens.

Human checkpoint: Validate the research before proceeding. Incorrect research cascades into broken plans and broken code.
Phase 2Research file as input

Plan

Outline every implementation step with explicit file names, function signatures, and code snippets. Plans with actual code are more reliable than prose descriptions.

1

Load the research file

Use the compressed research as the context seed — not the raw codebase.

2

List steps with artifacts

Each step should name the exact file to edit, the function to change, and a short code snippet showing the intended change.

3

Get the plan reviewed

Plans double as mental alignment docs for the whole team — they should be readable without running the code.

Plans with explicit snippets reduce implementation errors by giving the model a precise target, not a vague description to interpret.
Phase 3Plan file as input

Implement

Execute the plan in small increments. Validate continuously — don't run 10 steps and check at the end.

1

Work step by step

Follow the plan exactly. If something diverges, stop and update the plan rather than improvising.

2

Run tests between steps

Add test and build output back into the context as compressed summaries — not full logs.

3

Stay in the smart zone

If the context grows too large, compact: summarise earlier exchanges and remove resolved steps from the plan.

When implementation diverges from the plan without human review, AI output quality degrades fast. Don't outsource the thinking.

When to skip the full workflow

Not every task needs three phases. Use direct prompting for:

Minor UI tweaks Single-function fixes Greenfield features in isolation Well-understood, small-scope changes

Reserve the full workflow for complex, multi-file, brownfield tasks where incorrect assumptions are expensive.

Core principle

Agents are software. Not autonomous entities — they are deterministic programs with owned control flow, managed state, and carefully engineered prompts. The "magic" is just structured JSON flowing into code.

The 12 factors — click to expand

LLM output as structured data

The most powerful use of an LLM is turning natural language into structured JSON. That JSON is then processed by regular deterministic code — loops, switches, conditionals. Tool use is just passing JSON to a function.

Reframe "tool use" as: LLM outputs a JSON object → your code routes it to the right handler. No mysticism, just a function call.

Micro-agents and modular workflows

Break large workflows into small agents with 3–10 steps each. A deployment pipeline might have deterministic code handling CI/CD steps, and a micro-agent handling natural-language decisions like "should we roll back?"

Monolithic loop

One agent, 40+ steps, tangled context, hard to debug, failure cascades.

Micro-agents

Small focused agents, 3–10 steps, composable, isolated context, easy to test and replace.
